# backend

All server-side code: Lambda function handlers, shared middleware, AI agents, and database access.

## Why Lambda

Lambda was chosen over a traditional server (EC2, ECS, App Runner) because:

- **Cost** — pay per request, not per hour. At side-project scale the cost is effectively zero.
- **Ops** — no server to patch, no containers to manage.
- **Scaling** — handles traffic spikes without configuration.
- **Simplicity** — each handler is a plain TypeScript function with a clear input/output shape.

The trade-off is cold starts (~200–400ms on first request after idle). For a cooking app this is acceptable.

## Structure

```
backend/
├── functions/          # Lambda handlers — one file per domain
│   └── recipes.ts      # CRUD for recipes + URL import
├── middleware/          # Shared utilities used by all handlers
│   ├── auth.ts          # Cognito JWT validation
│   ├── errors.ts        # Error classes and response shaping
│   └── validation.ts    # Zod request body parsing
├── agents/             # AI agent modules (see agents/README.md)
├── db/
│   ├── client.ts        # Neon + Drizzle client (singleton, lazy-initialised)
│   ├── schema/          # Drizzle table definitions
│   └── queries/         # Typed query helper functions
├── triggers/
│   └── cognito-post-confirmation.ts  # Runs after a user confirms their email
└── scripts/
    └── bundle-lambdas.mjs  # esbuild bundler — produces dist/lambda/<name>.js
```

## How a request flows

```
API Gateway → Lambda → validateAuth() → handler logic → DB query → response
```

1. **API Gateway** (HTTP API v2) receives the request and routes it to the correct Lambda based on method + path
2. **`validateAuth()`** in `middleware/auth.ts` extracts and verifies the Cognito JWT from the `Authorization` header using `aws-jwt-verify`
3. The handler calls `getUserByCognitoId()` to resolve the Cognito sub claim to an internal user UUID
4. Business logic runs — Zod validates the request body, query helpers interact with the DB
5. Response is returned in the standard envelope: `{ data: ... }` on success, `{ error: { code, message } }` on failure

## Middleware

### `auth.ts`
Validates the Cognito ID token on every protected request. Uses `amazon-cognito-jwt-verify` for offline validation — no network call to Cognito on each request. Accepts tokens from multiple client IDs (web + mobile) via the `COGNITO_CLIENT_IDS` environment variable (comma-separated).

### `errors.ts`
Defines error classes (`NotFoundError`, `ValidationError`, `AuthError`) and a `handleError()` function that maps them to the correct HTTP status codes. Keeps handler code clean — just `throw new NotFoundError(...)`.

### `validation.ts`
Wraps Zod's `parse()` with a try/catch that throws a `ValidationError` (mapped to 422) on failure. All request body validation goes through this.

## Database

**Neon** (serverless Postgres) via **Drizzle ORM**.

### Why Neon
- Serverless Postgres that also scales to zero — no idle cost
- Branching (useful for staging in future)
- Compatible with standard Postgres tooling

### Why Drizzle (over Prisma)
- Schema is plain TypeScript — no separate `.prisma` DSL to learn
- Generated queries are readable SQL with no magic
- Lighter bundle size, important for Lambda cold starts
- Type inference from schema means types and DB stay in sync automatically

### Schema
Tables are defined in `db/schema/`. Each table file is imported and re-exported from `db/schema/index.ts`.

### Queries
Reusable query functions live in `db/queries/`. Handlers call these functions rather than building queries inline. This keeps handlers thin and makes the DB logic independently testable.

### Migrations
Generated by `drizzle-kit` and stored in `migrations/`. Run against the database manually for now:

```bash
DATABASE_URL=<neon-connection-string> npx drizzle-kit migrate
```

This will be automated as part of the deployment pipeline once the schema is more stable.

## Bundling

Each Lambda is bundled by esbuild into a single `.js` file in `dist/lambda/`. The bundler entry points are defined in `scripts/bundle-lambdas.mjs`. To add a new Lambda, add an entry there and a corresponding Terraform resource.

```bash
# Bundle all Lambdas
pnpm bundle:lambdas
```

esbuild tree-shakes unused code and inlines all dependencies except `@aws-sdk/*`, which is provided by the Lambda runtime.

## Environment variables

| Variable | Purpose |
|---|---|
| `DATABASE_URL` | Neon Postgres connection string |
| `NODE_ENV` | `production` in deployed environments |
| `COGNITO_USER_POOL_ID` | For JWT validation |
| `COGNITO_CLIENT_IDS` | Comma-separated list of allowed Cognito client IDs |

Sensitive values are passed via Terraform from GitHub Actions secrets — never committed.

## Running tests

```bash
pnpm --filter @souschef/backend test
```

Tests use **Vitest** and mock the DB client and AWS SDK. Each test file lives next to its source file.
